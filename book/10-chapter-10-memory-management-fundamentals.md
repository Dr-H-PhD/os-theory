# Chapter 10: Memory Management Fundamentals

Every instruction a processor executes, every variable a program reads, every stack frame a function creates --- all of these require memory. The operating system's memory management subsystem is the gatekeeper that decides where each byte lives, which process can touch which address, and what happens when physical memory runs short. Get this wrong, and programs crash, corrupt one another's data, or grind to a halt. Get it right, and dozens of processes coexist peacefully, each believing it has the machine to itself.

This chapter builds the foundations of memory management from first principles. We begin with the distinction between logical and physical addresses, work through the hardware and software mechanisms that bind one to the other, and then analyse the allocation strategies that determine how memory is parcelled out to competing processes. Along the way, we encounter fragmentation --- the silent tax that every allocator must pay --- and swapping, the brute-force escape hatch for when physical memory is exhausted.

The ideas in this chapter predate virtual memory. They represent the techniques used in early batch and multiprogramming systems and remain relevant today because they underpin the more sophisticated schemes of Chapters 11--13.

## 10.1 Address Spaces

### 10.1.1 Logical and Physical Addresses

A central abstraction in memory management is the distinction between the addresses a program uses and the addresses the hardware presents to the memory bus.

::: definition
**Definition 10.1 (Logical Address).** A *logical address* (also called a *virtual address*) is an address generated by the CPU during program execution. It is the address that appears in machine instructions, pointer variables, and program counters. The set of all logical addresses used by a process is its *logical address space*.
:::

::: definition
**Definition 10.2 (Physical Address).** A *physical address* is an address that appears on the memory bus and is understood by the memory controller. It identifies an actual location in the physical DRAM or SRAM. The set of all physical addresses available to the system is the *physical address space*.
:::

In the simplest systems --- a bare-metal microcontroller with no operating system --- logical and physical addresses are identical. The address 0x2000 in the program refers to byte 0x2000 in the physical memory chip. This works when only one program runs at a time and when the program is always loaded at the same address.

The moment we want to run multiple programs, or load a program at an arbitrary address, we need a mapping from logical to physical addresses. This mapping is the core responsibility of the memory management unit (MMU), a hardware component that sits between the CPU and the memory bus.

::: definition
**Definition 10.3 (Memory Management Unit).** The *MMU* is a hardware device that translates every logical address issued by the CPU into a physical address before it reaches the memory bus. The translation is governed by tables and registers maintained by the operating system.
:::

The relationship between logical and physical addresses depends on when the binding between them is established. Before we investigate binding, we must understand the size of address spaces.

### 10.1.2 Address Space Size

For a processor with an $n$-bit address bus, the logical address space contains $2^n$ bytes. A 32-bit processor can address $2^{32} = 4{,}294{,}967{,}296$ bytes (4 GB). A 64-bit processor can theoretically address $2^{64}$ bytes (16 exabytes), but current implementations use only 48 or 57 bits of the virtual address, yielding 256 TB or 128 PB of virtual address space respectively.

The physical address space is determined by the width of the physical address bus, which may differ from the logical address width. A 32-bit processor might have a 36-bit physical address bus (Intel PAE), giving 64 GB of physical memory. Conversely, a 64-bit processor with 48-bit virtual addresses might have only 46 bits of physical address, giving 64 TB of physical addressability.

::: example
**Example 10.1 (Address Space Arithmetic).** A system has a 32-bit logical address and a 28-bit physical address.

- Logical address space: $2^{32} = 4$ GB
- Physical address space: $2^{28} = 256$ MB

Each process believes it has 4 GB of memory, but the total physical memory is only 256 MB. The operating system must manage this disparity through address translation and, when necessary, swapping or demand paging.
:::

::: example
**Example 10.2 (Modern x86-64 Address Spaces).** An Intel Xeon processor implements 48-bit virtual addresses and 46-bit physical addresses.

- Virtual address space per process: $2^{48} = 256$ TB
- Physical address space: $2^{46} = 64$ TB
- Typical installed RAM: 256 GB $= 2^{38}$ bytes

The ratio of virtual to physical is $2^{48} / 2^{38} = 1024$. Each process has a virtual address space 1024 times larger than the installed physical memory. The operating system uses demand paging to maintain the illusion that all 256 TB is available.
:::

### 10.1.3 The Address Space as an Abstraction

The address space is not merely a bookkeeping device --- it is one of the most powerful abstractions in computer science. It provides three essential properties:

1. **Isolation:** Each process has its own address space. Process A's address 0x4000 and Process B's address 0x4000 refer to different physical locations. Neither process can access the other's memory (unless explicitly shared).

2. **Transparency:** The program does not know (or need to know) which physical addresses it occupies. It can be compiled once and run at any physical location.

3. **Overcommitment:** The total virtual memory across all processes can far exceed physical memory. The OS manages the illusion through demand paging and swapping.

::: definition
**Definition 10.4 (Address Space Isolation).** Two address spaces $A_1$ and $A_2$ are *isolated* if no logical address in $A_1$ maps to the same physical address as any logical address in $A_2$, unless both mappings are explicitly established by the operating system for the purpose of sharing.
:::

## 10.2 Address Binding

The process of associating a logical address with a physical address is called *address binding*. It can occur at three stages in the lifecycle of a program. Understanding these stages is essential because each one offers different trade-offs between flexibility, performance, and complexity.

### 10.2.1 Compile-Time Binding

If the physical memory location where a program will reside is known at compile time, the compiler generates code with absolute physical addresses embedded directly in the instructions.

::: definition
**Definition 10.5 (Compile-Time Binding).** *Compile-time binding* occurs when the compiler generates absolute addresses in the object code. The logical addresses in the program are identical to the physical addresses where the program will be loaded.
:::

::: example
**Example 10.3 (MS-DOS .COM Programs).** Early MS-DOS .COM executables used compile-time binding. The program was always loaded at address 0x0100 in the segment, and the compiler generated all addresses relative to this fixed base. If the program needed to be loaded elsewhere, the source code had to be recompiled with a different base address.
:::

::: example
**Example 10.4 (Embedded System).** Consider a traffic light controller running on an ARM Cortex-M0 microcontroller with 16 KB of flash at address 0x08000000. The linker script fixes the text segment at 0x08000000 and the data segment at 0x20000000 (SRAM). Every pointer in the compiled binary contains these absolute addresses. There is no MMU, no OS, and no possibility of loading the program elsewhere.
:::

Compile-time binding is rigid: if the load address changes, the entire program must be recompiled. This is acceptable only when the system runs a single program at a fixed location --- a common scenario in embedded systems but untenable for general-purpose operating systems.

### 10.2.2 Load-Time Binding

If the load address is not known at compile time, the compiler generates *relocatable code* containing relative addresses. When the loader places the program into memory, it adjusts all addresses by adding the actual base address.

::: definition
**Definition 10.6 (Load-Time Binding).** *Load-time binding* occurs when the linker/loader resolves logical addresses to physical addresses at the moment the program is loaded into memory. The compiler generates relocatable addresses, and the loader adds the base address to produce physical addresses.
:::

The relocatable code contains a *relocation table* --- a list of all locations in the binary that contain addresses needing adjustment. The loader iterates through this table, adding the base address to each entry.

::: example
**Example 10.5 (Relocation Table).** Consider a program compiled with a base of 0. It contains an instruction `LOAD R1, [0x0040]` (load from address 0x40). The relocation table records that the address operand at binary offset 0x0012 refers to logical address 0x0040.

If the loader places the program at physical address 0x5000, it rewrites the instruction to `LOAD R1, [0x5040]` by adding 0x5000 to the address operand. Every entry in the relocation table is similarly adjusted.
:::

::: example
**Example 10.6 (Relocation Cost).** A program with 10,000 relocatable addresses is loaded at address 0x80000. The loader must:

1. Read the relocation table (10,000 entries).
2. For each entry, read the current value at the specified offset, add 0x80000, and write back.
3. Total: 10,000 read-modify-write operations.

On a modern system, this takes microseconds. On early systems with slow disks, it was a noticeable delay.
:::

Load-time binding is more flexible than compile-time binding because the program can be loaded at any address. However, once loaded, the program cannot move without repeating the relocation process. This limitation motivates execution-time binding.

### 10.2.3 Execution-Time Binding

The most flexible approach defers address translation to the moment each instruction accesses memory. The program uses logical addresses throughout its execution, and the hardware translates them to physical addresses on every memory reference.

::: definition
**Definition 10.7 (Execution-Time Binding).** *Execution-time binding* (also called *dynamic binding* or *run-time binding*) occurs when address translation is performed by the MMU on every memory access during program execution. The program retains its logical addresses; the mapping to physical addresses can change without modifying the program.
:::

Execution-time binding requires hardware support (the MMU) but provides critical capabilities:

- **Relocation**: A process can be moved in physical memory without changing its code. The OS simply updates the translation tables.
- **Protection**: The MMU can enforce access restrictions, preventing one process from reading or writing another's memory.
- **Sharing**: Multiple logical addresses (in different processes) can map to the same physical address, enabling shared libraries and inter-process communication.
- **Demand paging**: Pages can be loaded from disk as needed, because the MMU can detect accesses to pages not yet in memory.

::: theorem
**Theorem 10.1 (Binding Flexibility Hierarchy).** Let $F_c$, $F_l$, and $F_e$ denote the sets of memory configurations achievable with compile-time, load-time, and execution-time binding, respectively. Then $F_c \subseteq F_l \subseteq F_e$. The inclusions are strict whenever the system supports more than one process.

*Proof.* Every configuration achievable by compile-time binding (fixed addresses) is trivially achievable by load-time binding (set the base to zero). Every configuration achievable by load-time binding is achievable by execution-time binding (set the MMU translation to the loader's base offset). The first inclusion is strict because load-time binding can place a program at an arbitrary address, which compile-time binding cannot (unless recompiled). The second inclusion is strict because execution-time binding permits relocation of a running process --- moving its physical pages while preserving logical addresses --- which load-time binding cannot achieve without halting and re-relocating the process. $\square$
:::

### 10.2.4 Dynamic Loading and Linking

Two related concepts extend the binding discussion:

**Dynamic loading** defers loading of a module until it is first called. The main program is loaded; subroutines remain on disk until invoked. This reduces memory consumption for programs with large libraries where only a fraction of routines are used in any given run.

**Dynamic linking** defers the linking of a shared library until load time or run time. Instead of including library code in the executable, the program contains *stubs* --- small pieces of code that locate and call the actual library function. On the first call, the stub is replaced with the address of the real function. Subsequent calls go directly to the library.

::: definition
**Definition 10.8 (Dynamic Linking).** *Dynamic linking* is a mechanism where references to shared library functions are resolved at load time or at first use (lazy binding). The executable contains *stubs* that, on first invocation, invoke the dynamic linker to resolve the symbol and patch the calling code.
:::

On Linux, the dynamic linker is `ld-linux.so`. The Procedure Linkage Table (PLT) and Global Offset Table (GOT) implement the stub mechanism. Lazy binding means that the cost of resolving a symbol is paid only when the function is first called, not when the library is loaded.

::: example
**Example 10.7 (Dynamic Linking on Linux).** When a C program calls `printf`, the following happens on the first call:

1. The call instruction jumps to the PLT entry for `printf`.
2. The PLT stub reads the GOT entry, which initially points back to the PLT stub's "resolver" code.
3. The resolver invokes `ld-linux.so`, which searches the loaded libraries for the symbol `printf`.
4. `ld-linux.so` finds `printf` in `libc.so` at address 0x7f4a38201a40 and writes this address into the GOT.
5. The resolver jumps to `printf`.
6. On subsequent calls, the PLT stub reads the GOT entry (now 0x7f4a38201a40) and jumps directly to `printf`, bypassing the resolver.
:::

## 10.3 Base and Limit Registers

The simplest hardware mechanism for execution-time address translation uses two registers: a *base register* and a *limit register*.

### 10.3.1 Mechanism

::: definition
**Definition 10.9 (Base-Limit Address Translation).** Given a base register $B$ and a limit register $L$:

1. For every logical address $a$ generated by the CPU, the MMU first checks $0 \leq a < L$.
2. If the check fails, the MMU raises a hardware trap (memory protection fault).
3. If the check passes, the MMU computes the physical address as $p = B + a$.
:::

The base register defines where the process begins in physical memory. The limit register defines the size of the address space. Together, they confine the process to a contiguous region $[B, B + L)$ in physical memory.

```text
Logical Address (from CPU)
        |
        v
   +---------+       +---------+
   | a < L ? |--NO-->|  TRAP   |
   +---------+       +---------+
        |
       YES
        |
        v
   +---------+
   | p = B+a |
   +---------+
        |
        v
Physical Address (to memory bus)
```

The entire translation adds a single comparison and a single addition to every memory access. On a pipelined processor, this can be overlapped with other stages, adding zero effective latency in the common case.

### 10.3.2 Context Switching

When the operating system switches from one process to another, it must save the current process's base and limit values and load the new process's values. This is part of the *context switch* overhead.

The base and limit registers are *privileged*: only the operating system (running in kernel mode) can modify them. A user-mode process cannot change its own base or limit, which prevents it from escaping its allocated region.

::: definition
**Definition 10.10 (Privileged Registers).** The base and limit registers are *privileged* (or *protected*) --- they can be modified only by instructions executed in kernel mode (ring 0 on x86). Any attempt to modify them from user mode raises a general protection fault.
:::

### 10.3.3 Worked Examples

::: example
**Example 10.8 (Two-Process Base-Limit).** Suppose the operating system has loaded Process A at physical address 0x10000 with size 0x8000, and Process B at 0x20000 with size 0xC000.

| Process | Base Register | Limit Register | Physical Range |
|---------|---------------|----------------|----------------|
| A | 0x10000 | 0x8000 | 0x10000--0x17FFF |
| B | 0x20000 | 0xC000 | 0x20000--0x2BFFF |

When Process A generates logical address 0x3000, the MMU computes: $0\text{x}3000 < 0\text{x}8000$ (pass), physical address $= 0\text{x}10000 + 0\text{x}3000 = 0\text{x}13000$.

When Process A generates logical address 0x9000, the MMU detects: $0\text{x}9000 \not< 0\text{x}8000$ (fail), and raises a segmentation fault.
:::

::: example
**Example 10.9 (Base-Limit with Multiple Processes).** A system has 1 MB of physical memory (0x00000 -- 0xFFFFF). The OS kernel occupies 0x00000 -- 0x0FFFF (64 KB). Three user processes are loaded:

| Process | Base | Limit | Physical Range |
|---------|------|-------|----------------|
| P1 | 0x10000 | 0x20000 | 0x10000 -- 0x2FFFF |
| P2 | 0x30000 | 0x40000 | 0x30000 -- 0x6FFFF |
| P3 | 0x70000 | 0x10000 | 0x70000 -- 0x7FFFF |

Free memory: 0x80000 -- 0xFFFFF (512 KB).

If P2 terminates, the free memory becomes: 0x30000 -- 0x6FFFF (256 KB) and 0x80000 -- 0xFFFFF (512 KB). A new process requesting 300 KB cannot use the first hole (only 256 KB) and would be placed in the second hole.
:::

### 10.3.4 Relocation with Base-Limit

Because addresses are translated dynamically, the operating system can *relocate* a process by:

1. Suspending the process.
2. Copying its memory contents from $[B_{\text{old}}, B_{\text{old}} + L)$ to $[B_{\text{new}}, B_{\text{new}} + L)$.
3. Updating the base register to $B_{\text{new}}$.
4. Resuming the process.

The process is unaware of the move because its logical addresses have not changed. This ability to relocate is essential for *compaction*, discussed in Section 10.6.

### 10.3.5 Limitations of Base-Limit

The base-limit scheme has several significant limitations:

1. **Contiguity requirement:** Each process must occupy a contiguous block of physical memory. This leads to external fragmentation.
2. **No sharing:** Two processes cannot share code or data, because each process's address space is a single contiguous block with no overlap.
3. **No fine-grained protection:** Every address in the range $[0, L)$ has the same read/write/execute permissions. You cannot make the code segment read-only while keeping the data segment read-write.
4. **Entire process must be in memory:** There is no mechanism for keeping only part of the process in physical memory.

These limitations motivate the more sophisticated mechanisms of segmentation and paging.

## 10.4 Contiguous Memory Allocation

In a contiguous allocation scheme, each process occupies a single contiguous block of physical memory. The operating system must decide how to divide physical memory among competing processes.

### 10.4.1 Fixed Partitions

The simplest approach divides memory into a fixed number of partitions at system startup.

::: definition
**Definition 10.11 (Fixed Partition Allocation).** In *fixed partition allocation*, physical memory is divided into $k$ partitions of predetermined sizes $s_1, s_2, \ldots, s_k$ at boot time. Each incoming process is assigned to a partition large enough to hold it.
:::

Two queuing strategies are possible:

**One queue per partition.** Each partition has its own ready queue. An incoming process of size $n$ is placed in the queue of the smallest partition with $s_i \geq n$. This can lead to underutilisation if one partition's queue is empty while another is full.

**Single shared queue.** All processes share one queue. When a partition becomes free, the next process in the queue that fits is assigned to it.

::: example
**Example 10.10 (Fixed Partitions).** A system has 64 MB of memory divided into four fixed partitions of sizes 8 MB, 16 MB, 16 MB, and 24 MB. Three processes arrive with sizes 7 MB, 14 MB, and 20 MB.

| Process | Size | Assigned Partition | Partition Size | Internal Fragmentation |
|---------|------|--------------------|----------------|------------------------|
| P1 | 7 MB | Partition 1 | 8 MB | 1 MB |
| P2 | 14 MB | Partition 2 | 16 MB | 2 MB |
| P3 | 20 MB | Partition 4 | 24 MB | 4 MB |

Total memory used: 41 MB out of 64 MB. Internal fragmentation: 7 MB. Partition 3 (16 MB) is entirely unused.
:::

::: example
**Example 10.11 (Fixed Partition Scheduling Problem).** Same system as above. A fourth process P4 of size 15 MB arrives. Under single-queue scheduling, P4 would fit in Partition 3 (16 MB). Under per-partition queuing, P4 joins the queue for the 16 MB partition (Partition 2 or 3) and must wait even though Partition 4 (24 MB) might be free.

This illustrates the *scheduling anomaly* of per-partition queuing: a process may wait in a queue while a larger partition sits idle. The single-queue approach avoids this but requires the OS to search for the best-fit partition for each process.
:::

Fixed partitions are easy to implement but inflexible. The partition sizes are chosen at boot time and cannot adapt to workload changes. A system designed for a mix of small and large processes may perform poorly when the workload shifts.

::: theorem
**Theorem 10.2 (Maximum Utilisation of Fixed Partitions).** For a system with $k$ partitions of sizes $s_1 \leq s_2 \leq \cdots \leq s_k$ and a stream of processes whose sizes are uniformly distributed on $[0, s_k]$, the expected memory utilisation is:

$$U = \frac{1}{k} \sum_{i=1}^{k} \left(1 - \frac{s_i}{2 \cdot s_i}\right) = \frac{1}{k} \sum_{i=1}^{k} \frac{1}{2} = \frac{1}{2}$$

That is, on average, half the allocated memory is wasted as internal fragmentation. This bound is tight when the partitions are well-sized for the workload.
:::

### 10.4.2 Variable Partitions

A more flexible approach allows partitions to be created dynamically, sized exactly to the process's needs.

::: definition
**Definition 10.12 (Variable Partition Allocation).** In *variable partition allocation*, each process is allocated a contiguous block of memory exactly equal to its size (or slightly larger due to alignment). Partitions are created and destroyed as processes arrive and depart.
:::

The operating system maintains a data structure (typically a linked list or a set of lists) tracking the free regions of memory, called *holes*.

::: example
**Example 10.12 (Variable Partition Dynamics).** Consider a system with 64 MB of memory. Initially, the entire memory is one free hole. Processes arrive and depart over time:

1. **P1 (10 MB) arrives:** Allocated at 0. Free: 10--63 (54 MB).
2. **P2 (20 MB) arrives:** Allocated at 10. Free: 30--63 (34 MB).
3. **P3 (15 MB) arrives:** Allocated at 30. Free: 45--63 (19 MB).
4. **P2 (20 MB) departs:** Free holes: 10--29 (20 MB) and 45--63 (19 MB).
5. **P4 (8 MB) arrives:** Allocated at 10 (first-fit). Free: 18--29 (12 MB) and 45--63 (19 MB).

After step 5, memory has three regions: P1 (0--9), P4 (10--17), free (18--29), P3 (30--44), free (45--63). The free memory is split into two non-contiguous holes.
:::

### 10.4.3 Free List Data Structures

The data structure used to maintain the free list significantly affects allocation performance. The choice of data structure is itself an engineering trade-off between allocation speed, deallocation speed (including coalescing), and memory overhead.

**Linked list (address-ordered).** The simplest implementation: each hole is a node in a doubly-linked list, sorted by address. Allocation requires scanning the list; deallocation requires finding the correct position and potentially merging with adjacent free blocks.

- Allocation (first-fit): $O(n)$ scan until the first adequate hole.
- Deallocation: $O(n)$ to find the position by address, then $O(1)$ to merge with neighbours.

::: example
**Example 10.12a (Free List Coalescing).** The free list (sorted by address) contains: [10--19], [30--49], [70--99]. Process B (occupying 20--29) terminates.

1. Insert hole [20--29] into the list between [10--19] and [30--49].
2. Check left neighbour [10--19]: contiguous (end=19, new start=20). Merge: [10--29].
3. Check right neighbour [30--49]: contiguous (new end=29, right start=30). Merge: [10--49].
4. Result: [10--49], [70--99].

Two merges reduced three holes to one. Without coalescing, the free list would have four holes: [10--19], [20--29], [30--49], [70--99].
:::

**Segregated free lists.** Maintain separate free lists for different size ranges: one list for holes $< 4$ KB, one for 4--16 KB, one for 16--64 KB, etc. Allocation first checks the appropriate list, falling back to larger lists if necessary. This reduces average search time because only a subset of holes is examined.

::: definition
**Definition 10.12a (Segregated Free List).** A *segregated free list* organises holes into $k$ size classes $[0, s_1), [s_1, s_2), \ldots, [s_{k-1}, \infty)$. Each class has its own linked list. An allocation of size $n$ first checks the list for the smallest class containing holes $\geq n$. If empty, the next larger class is checked, and so on.
:::

**Balanced tree (by size).** An AVL tree or red-black tree keyed by hole size allows best-fit allocation in $O(\log n)$ time. This is used in some advanced allocators but adds complexity for insertion and deletion.

**Bitmap.** Divide memory into fixed-size units (e.g., 1 KB). Maintain a bitmap where bit $i$ = 1 if unit $i$ is allocated, 0 if free. Allocation: scan the bitmap for a run of $k$ consecutive zeros. Deallocation: set the corresponding bits to 0. Space: $M / U$ bits where $M$ is total memory and $U$ is the unit size.

::: example
**Example 10.12b (Bitmap Allocation).** Memory: 16 units. Bitmap: 1111000011100010.

Free regions: units 4--7 (4 units), units 11--11 (1 unit), units 13--14 (2 units), unit 15 is at end but bit=0. Wait, reading left to right: bits 0--3 = 1111 (allocated), bits 4--7 = 0000 (free), bits 8--10 = 111 (allocated), bit 11 = 0 (free), bit 12 = 0 (free), bit 13 = 0 (free), bit 14 = 1 (allocated), bit 15 = 0 (free).

Free regions: [4--7] (4 units), [11--13] (3 units), [15] (1 unit).

Request for 3 units (first-fit): allocate units 4--6. New bitmap: 1111111011100010.
:::

### 10.4.4 Coalescing Policies

When a block is freed, adjacent free blocks should be merged to prevent the proliferation of small, unusable holes. Two coalescing policies exist:

**Immediate coalescing:** Every deallocation immediately checks for and merges adjacent free blocks. This keeps the free list clean but adds overhead to every deallocation.

**Deferred coalescing:** Freed blocks are not immediately merged. Coalescing is triggered periodically or when an allocation request cannot be satisfied. This reduces per-deallocation overhead but allows temporary fragmentation.

::: theorem
**Theorem 10.2a (Coalescing Reduces Fragmentation).** In a system with $n$ allocations and $n$ deallocations (in any order), the number of free holes at any time is at most $n$ without coalescing. With immediate coalescing, the number of holes is bounded by $\lceil n/2 \rceil$ (by the fifty-percent rule). The total size of the largest free block is always at least as large with coalescing as without.

*Proof sketch.* Each deallocation creates at most one new hole. Without coalescing, $n$ deallocations create at most $n$ holes. With coalescing, each deallocation may merge with up to two neighbours, reducing the hole count. The merged hole is always at least as large as the sum of its constituent holes, so the largest hole size is non-decreasing under coalescing. $\square$
:::

## 10.5 Memory Allocation Strategies

When a process requests $n$ bytes and multiple free holes can satisfy the request, the operating system must choose which hole to use. Three classical strategies exist.

### 10.5.1 First-Fit

::: definition
**Definition 10.13 (First-Fit).** *First-fit* allocation scans the list of free holes from the beginning and allocates the first hole that is large enough.
:::

First-fit is fast because it stops as soon as it finds a suitable hole. It tends to leave small fragments near the beginning of the list (where allocations happen frequently) and larger holes towards the end.

**Complexity:** $O(n)$ in the worst case, where $n$ is the number of free holes. With optimisations (e.g., a roving pointer that starts searching where the last allocation ended, called *next-fit*), average-case performance improves.

### 10.5.2 Next-Fit

::: definition
**Definition 10.14 (Next-Fit).** *Next-fit* is a variant of first-fit that begins the search at the position where the previous allocation ended, wrapping around to the beginning if necessary. This distributes allocations more evenly across memory.
:::

Next-fit avoids the clustering of small fragments at the beginning of the free list that plagues first-fit. However, simulations show that next-fit often produces worse overall fragmentation than first-fit because it tends to break up the large holes at the end of memory that first-fit preserves.

### 10.5.3 Best-Fit

::: definition
**Definition 10.15 (Best-Fit).** *Best-fit* allocation searches the entire list of free holes and selects the smallest hole that is large enough to satisfy the request.
:::

Best-fit minimises the *leftover* from each allocation, producing the smallest possible residual hole. Paradoxically, this often leads to worse overall fragmentation because the tiny leftover holes are too small to be useful for future allocations.

**Complexity:** $O(n)$ to scan the entire free list. Can be reduced to $O(\log n)$ with a balanced tree sorted by hole size.

### 10.5.4 Worst-Fit

::: definition
**Definition 10.16 (Worst-Fit).** *Worst-fit* allocation searches the entire list of free holes and selects the largest hole.
:::

Worst-fit maximises the leftover, hoping that the residual hole will be large enough to satisfy future requests. In practice, it tends to produce mediocre results because it rapidly breaks large holes into medium-sized ones that may not serve any request well.

**Complexity:** $O(n)$ to scan the entire list, or $O(1)$ with a max-heap.

### 10.5.5 Comparative Analysis

::: theorem
**Theorem 10.3 (First-Fit and Best-Fit Dominance).** Simulation studies and analytical results consistently show that first-fit and best-fit outperform worst-fit in terms of both memory utilisation and allocation speed. Between the two, first-fit is generally preferred because it is faster (no need to scan the entire list) and produces comparable utilisation.

*Justification.* Knuth's analysis (1973) of the first-fit and best-fit algorithms using the *fifty-percent rule* shows that, under steady-state conditions with random request sizes, approximately one-third of the holes are unusable (external fragmentation). First-fit achieves this ratio while examining, on average, only a fraction of the free list. Best-fit achieves a slightly better ratio but at the cost of examining the entire list and creating more tiny, unusable fragments.
:::

::: example
**Example 10.13 (Strategy Comparison).** Free holes: 100 KB, 500 KB, 200 KB, 300 KB, 600 KB. Request: 212 KB.

| Strategy | Chosen Hole | Leftover | Remark |
|----------|-------------|----------|--------|
| First-fit | 500 KB | 288 KB | First hole $\geq$ 212 KB |
| Best-fit | 300 KB | 88 KB | Smallest adequate hole |
| Worst-fit | 600 KB | 388 KB | Largest hole |

First-fit allocates from the 500 KB hole (the first adequate one). Best-fit chooses the 300 KB hole, leaving an 88 KB fragment that may be too small for most requests. Worst-fit uses the 600 KB hole, preserving a large 388 KB residual.
:::

::: example
**Example 10.14 (Extended Comparison).** Consider a sequence of three requests applied to the same initial free list: 100 KB, 500 KB, 200 KB, 300 KB, 600 KB. Requests: 212 KB, then 100 KB, then 450 KB.

**First-fit trace:**

1. Request 212 KB: allocate from 500 KB hole. Free: 100, 288, 200, 300, 600.
2. Request 100 KB: allocate from 100 KB hole (exact fit). Free: 288, 200, 300, 600.
3. Request 450 KB: allocate from 600 KB hole. Free: 288, 200, 300, 150.

**Best-fit trace:**

1. Request 212 KB: allocate from 300 KB hole. Free: 100, 500, 200, 88, 600.
2. Request 100 KB: allocate from 100 KB hole. Free: 500, 200, 88, 600.
3. Request 450 KB: allocate from 500 KB hole. Free: 50, 200, 88, 600.

Best-fit leaves more holes (four vs four) but with smaller sizes (50 KB and 88 KB are likely unusable). First-fit's smallest hole is 150 KB, which may still be useful.
:::

### 10.5.6 The Fifty-Percent Rule

::: theorem
**Theorem 10.4 (Knuth's Fifty-Percent Rule).** In a system using first-fit allocation in steady state, if $H$ is the number of holes and $P$ is the number of allocated blocks, then:

$$H \approx \frac{P}{2}$$

*Proof.* Consider the steady-state condition where processes arrive and depart at equal rates. Each allocation either (a) splits a hole into an allocated block and a smaller hole (net change: $+1$ block, $0$ holes) or (b) exactly fills a hole ($+1$ block, $-1$ hole). Each deallocation either (a) creates a new hole adjacent to two allocated blocks ($+1$ hole) or (b) merges with one adjacent hole ($0$ net holes) or (c) merges with two adjacent holes ($-1$ hole).

Let $p_s$ be the probability that an allocation splits a hole (as opposed to filling it exactly); in practice $p_s \approx 1$. Let $p_0$, $p_1$, $p_2$ be the probabilities that a deallocation creates a new hole, merges with one, or merges with two neighbours.

In steady state, the expected change in the number of holes per operation must be zero. An allocation changes holes by $0$ (split) or $-1$ (exact fit). A deallocation changes holes by $+1$ (new), $0$ (merge with one), or $-1$ (merge with two).

If we consider a random configuration of $P$ allocated blocks and $H$ holes in a linear memory, the probability that a block's left neighbour is a hole is $H / (P + H)$, and similarly for the right neighbour (approximately, ignoring boundary effects). The expected number of holes merged on deallocation is:

$$E[\text{merges}] = 2 \times \frac{H}{P + H}$$

In steady state, the hole creation rate equals the hole consumption rate. Through careful bookkeeping (see Knuth, *The Art of Computer Programming*, Vol. 1, Section 2.5), this yields $H = P/2$ under the assumption that processes are randomly distributed in memory. $\square$
:::

This result has a practical implication: if a system has $P$ active processes, the memory will contain approximately $P/2$ free holes. Since each hole represents wasted (or at least fragmented) memory, the system is inherently inefficient --- motivating the compaction techniques of Section 10.6.

> **Programmer:** The fifty-percent rule explains why long-running C programs that perform many `malloc`/`free` cycles eventually suffer from fragmentation. Modern allocators (glibc's ptmalloc, jemalloc, TCMalloc) combat this with *size classes*: instead of managing one free list, they maintain separate free lists for common sizes (8, 16, 32, 64, ... bytes). This turns a variable-partition problem into a fixed-partition problem within each size class, eliminating external fragmentation at the cost of modest internal fragmentation. Go's memory allocator uses 67 size classes, and we shall see its full design in Chapter 13.

### 10.5.7 Formal Analysis of Allocation Strategies

We can formalise the allocation problem as follows. Let $\mathcal{H} = \{h_1, h_2, \ldots, h_k\}$ be the set of free holes with sizes $|h_1|, |h_2|, \ldots, |h_k|$. A request of size $n$ is *satisfiable* if $\exists i : |h_i| \geq n$.

::: definition
**Definition 10.17 (Allocation Function).** An *allocation function* $\alpha : (\mathcal{H}, n) \to \mathcal{H} \cup \{\bot\}$ maps a set of holes and a request size to either a selected hole or $\bot$ (failure). For first-fit, $\alpha$ returns the first $h_i$ (in address order) with $|h_i| \geq n$. For best-fit, $\alpha$ returns $\arg\min_{|h_i| \geq n} |h_i|$. For worst-fit, $\alpha$ returns $\arg\max_i |h_i|$ (if $|h_i| \geq n$).
:::

::: theorem
**Theorem 10.5 (No Optimal Online Strategy).** No online allocation strategy (one that does not know future requests) can guarantee the minimum number of processes that fail to be allocated. The competitive ratio of first-fit against the optimal offline algorithm is $O(\log n)$ for $n$ requests.

This is proved by reduction to the online bin-packing problem: allocating variable-sized processes into a fixed memory is equivalent to packing items into bins, and first-fit-decreasing achieves a competitive ratio of 11/9 in the offline case, but no online algorithm can do better than $\Omega(\log n)$ in the worst case.
:::

## 10.6 Fragmentation

Fragmentation is the phenomenon where available memory is wasted because it is divided into pieces too small or too awkwardly placed to satisfy allocation requests.

### 10.6.1 Internal Fragmentation

::: definition
**Definition 10.18 (Internal Fragmentation).** *Internal fragmentation* is the memory wasted inside an allocated block because the block is larger than the request. It occurs when memory is allocated in fixed-size units and the request does not evenly divide the unit size.
:::

::: example
**Example 10.15 (Internal Fragmentation).** A system allocates memory in 4 KB pages. A process requests 4,100 bytes. The system allocates 2 pages (8,192 bytes). The internal fragmentation is $8{,}192 - 4{,}100 = 4{,}092$ bytes --- nearly an entire page wasted.
:::

::: example
**Example 10.16 (Internal Fragmentation in Size Classes).** A size-class allocator has classes at 8, 16, 32, 64, 128, 256, 512, and 1024 bytes. A request for 65 bytes is rounded up to the 128-byte class. Internal fragmentation: $128 - 65 = 63$ bytes (49%).

A request for 33 bytes is rounded up to 64 bytes. Internal fragmentation: $64 - 33 = 31$ bytes (48%).

A request for 64 bytes is an exact fit. Internal fragmentation: 0 bytes (0%).
:::

The expected internal fragmentation per allocation, assuming requests are uniformly distributed in $[1, U]$ where $U$ is the allocation unit size, is:

$$E[\text{internal fragmentation}] = \frac{U - 1}{2}$$

This follows because the wasted space is uniformly distributed on $[0, U - 1)$.

### 10.6.2 External Fragmentation

::: definition
**Definition 10.19 (External Fragmentation).** *External fragmentation* is the condition where the total free memory is sufficient to satisfy a request, but no single contiguous block is large enough. The free memory is scattered across many small holes.
:::

::: example
**Example 10.17 (External Fragmentation).** A system has three free holes: 50 KB, 30 KB, and 40 KB (total: 120 KB). A process requests 100 KB. Although 120 KB is free, no single hole is large enough, so the request fails.
:::

::: example
**Example 10.18 (Progressive External Fragmentation).** A system with 256 KB of memory processes the following sequence:

1. Allocate A (80 KB) at 0. Free: 80--255 (176 KB).
2. Allocate B (60 KB) at 80. Free: 140--255 (116 KB).
3. Allocate C (40 KB) at 140. Free: 180--255 (76 KB).
4. Free A. Free: 0--79 (80 KB), 180--255 (76 KB).
5. Allocate D (50 KB) at 0 (first-fit). Free: 50--79 (30 KB), 180--255 (76 KB).
6. Free C. Free: 50--79 (30 KB), 140--179 (40 KB), 180--255 (76 KB).

Total free: 146 KB, in three holes. A request for 100 KB fails despite 146 KB being free.
:::

External fragmentation is the bane of variable-partition allocation. Over time, as processes of varying sizes arrive and depart, memory becomes a patchwork of allocated blocks and small holes.

### 10.6.3 Quantifying External Fragmentation

We can measure external fragmentation as:

$$F_{\text{ext}} = 1 - \frac{\text{largest free block}}{\text{total free memory}}$$

When $F_{\text{ext}} = 0$, all free memory is in one contiguous block (no external fragmentation). When $F_{\text{ext}}$ approaches 1, free memory is scattered into many tiny, unusable holes.

::: example
**Example 10.19 (Fragmentation Metric).** From Example 10.18, step 6: free holes of 30, 40, and 76 KB. Total free: 146 KB. Largest hole: 76 KB.

$$F_{\text{ext}} = 1 - \frac{76}{146} = 1 - 0.521 = 0.479$$

Nearly half the free memory is "lost" to fragmentation.
:::

::: theorem
**Theorem 10.6 (Fragmentation Bound).** Statistical analysis by Knuth and others shows that with first-fit allocation and random request sizes drawn from a distribution with mean $\mu$ and finite variance, the fraction of memory lost to external fragmentation converges to approximately $1/3$ in the steady state. More precisely, if $M$ is the total memory and $U$ is the total memory in use, then:

$$\frac{M - U}{M} \approx \frac{1}{3}$$

in the long run, under mild assumptions on the request size distribution.
:::

This "one-third rule" means that a system must have roughly 50% more memory than the total size of the processes it wishes to run simultaneously.

### 10.6.4 Compaction

::: definition
**Definition 10.20 (Compaction).** *Compaction* is the process of relocating allocated blocks to consolidate all free memory into one large contiguous hole. It is possible only if the system uses execution-time address binding (so that processes can be moved without rewriting their code).
:::

Compaction eliminates external fragmentation but has a significant cost: every byte of allocated memory may need to be copied, and all processes must be suspended during the operation (or a complex incremental scheme must be used).

```text
Before compaction:              After compaction:

+--------+                      +--------+
|  P1    |                      |  P1    |
+--------+                      +--------+
|  free  |                      |  P3    |
+--------+                      +--------+
|  P3    |                      |  P5    |
+--------+                      +--------+
|  free  |                      |        |
+--------+                      |  free  |
|  P5    |                      |        |
+--------+                      |        |
|  free  |                      +--------+
+--------+
```

The cost of compaction is $O(M)$ where $M$ is the total memory size, because in the worst case every byte must be copied. For a system with gigabytes of memory, this can take milliseconds to seconds --- an eternity for interactive or real-time applications.

::: theorem
**Theorem 10.7 (Optimal Compaction is NP-Hard).** Given an initial configuration of allocated blocks and free holes, finding the compaction strategy that minimises the total number of bytes moved is NP-hard (reduction from the 3-partition problem).

*Proof sketch.* We reduce from the NP-hard problem of partitioning a multiset of integers into triples with equal sums. Given an instance of 3-partition, construct a memory configuration where each integer corresponds to a process size and the "bins" correspond to target positions. Minimising bytes moved in compaction corresponds to finding a valid 3-partition. $\square$
:::

In practice, operating systems use simple heuristics: move all allocated blocks to one end of memory (e.g., the low end), which is suboptimal but easy to implement. The cost is at most $\sum_i s_i$ bytes moved (the total size of all allocated blocks).

> **Programmer:** Modern systems almost never perform compaction of physical memory. Instead, they use paging (Chapter 11) to eliminate external fragmentation entirely: since processes are divided into fixed-size pages, and any page can be placed in any frame, contiguity in physical memory is unnecessary. However, compaction does appear in other contexts. Java's G1 and ZGC collectors compact live objects to reduce fragmentation. At the kernel level, Linux's `kcompactd` daemon performs *memory compaction* --- not of process memory, but of free pages --- to create contiguous blocks needed for huge pages and DMA buffers. You can trigger compaction manually:
>
> ```text
> echo 1 > /proc/sys/vm/compact_memory
> ```
>
> And check the compaction status in `/proc/vmstat` (look for `compact_*` counters). The Go garbage collector uses a non-moving design, avoiding compaction entirely at the cost of some fragmentation, managed through the size-class system described in Chapter 13.

## 10.7 Swapping

When the total memory demands of all active processes exceed the available physical memory, the operating system can use *swapping* to temporarily move entire processes out of memory.

### 10.7.1 Basic Swapping

::: definition
**Definition 10.21 (Swapping).** *Swapping* is the mechanism by which the operating system moves an entire process's memory image between main memory (RAM) and a secondary storage device (the *backing store* or *swap space*). A process that has been moved to the backing store is said to be *swapped out*; one that has been brought back is *swapped in*.
:::

The basic swapping algorithm works as follows:

1. When a process must be loaded and no free memory is available, the scheduler selects a victim process currently in memory.
2. The victim's entire memory image is written to the backing store (*swap out*).
3. The new process is loaded into the freed memory (*swap in*).
4. The victim process is marked as "swapped out" in the process table.
5. When the victim is scheduled again, it must be swapped back in.

### 10.7.2 Backing Store

The backing store is a region of secondary storage (usually a disk partition or a file) reserved for swapped-out process images.

::: definition
**Definition 10.22 (Backing Store).** The *backing store* is a contiguous region of secondary storage large enough to hold the memory images of all processes that might be swapped out simultaneously. It must support fast, direct access to any process's image.
:::

The backing store must be at least as large as the total virtual memory of all swappable processes. On Linux, this is the *swap partition* (or *swap file*). The `mkswap` and `swapon` commands configure swap space.

::: example
**Example 10.20 (Swap Configuration on Linux).** Creating and enabling a 4 GB swap file:

```text
# Create the swap file
dd if=/dev/zero of=/swapfile bs=1M count=4096

# Set permissions (only root should access swap)
chmod 600 /swapfile

# Format as swap
mkswap /swapfile

# Enable the swap
swapon /swapfile

# Verify
swapon --show
# NAME      TYPE SIZE  USED PRIO
# /swapfile file   4G   0B   -2
```
:::

### 10.7.3 Cost of Swapping

The time to swap a process is dominated by the data transfer time. If a process occupies $S$ bytes and the disk transfer rate is $R$ bytes per second, the swap time is:

$$T_{\text{swap}} = \frac{S}{R}$$

For a complete swap (out and in), the total time is $2T_{\text{swap}}$.

::: example
**Example 10.21 (Swap Time Comparison Across Storage Technologies).**

A process occupies 100 MB. We compare three storage technologies:

| Storage | Transfer Rate | Swap-Out Time | Swap-In Time | Total Cycle |
|---------|--------------|---------------|--------------|-------------|
| HDD (7200 RPM) | 150 MB/s | 667 ms | 667 ms | 1,334 ms |
| SATA SSD | 500 MB/s | 200 ms | 200 ms | 400 ms |
| NVMe SSD | 3,500 MB/s | 29 ms | 29 ms | 57 ms |

Even with the fastest NVMe SSD, swapping a 100 MB process takes 57 ms --- enough to cause a noticeable stutter in an interactive application refreshing at 60 Hz (16.7 ms per frame).
:::

::: example
**Example 10.22 (Swap Time with Seek Latency).** For a spinning HDD, the swap time includes seek time and rotational latency in addition to transfer time:

$$T_{\text{swap}} = T_{\text{seek}} + T_{\text{rotate}} + \frac{S}{R}$$

With $T_{\text{seek}} = 5$ ms, $T_{\text{rotate}} = 4$ ms (average for 7200 RPM), $R = 150$ MB/s, and $S = 100$ MB:

$$T_{\text{swap}} = 5 + 4 + \frac{100}{0.15} = 9 + 667 = 676 \text{ ms}$$

The seek and rotational latencies are negligible for large transfers but significant for small ones (swapping a single 4 KB page: $5 + 4 + 0.027 \approx 9$ ms, almost entirely latency).
:::

### 10.7.4 Swapping Policy

Not all processes are equally good candidates for swapping. The OS considers:

- **Process state:** A process blocked on I/O should not be swapped out, because the I/O buffer may be in its address space (DMA writes to physical memory).
- **Priority:** Low-priority processes are preferred victims.
- **Time in memory:** A process that was recently swapped in should not be immediately swapped out again (hysteresis).
- **Size:** Smaller processes are cheaper to swap (less data to transfer).

### 10.7.5 Partial Swapping and Paging

The major drawback of basic swapping is its all-or-nothing nature: the entire process must be moved, even if only a small portion of its memory is actively used. This observation motivates *demand paging* (Chapter 11), which swaps individual pages rather than entire processes. Demand paging is the dominant memory management strategy in all modern operating systems.

> **Note:** Modern operating systems (Linux, Windows, macOS) no longer perform process-level swapping in the traditional sense. Instead, they use demand paging with page-level swap. The kernel writes individual dirty pages to swap space and reads them back on demand. The term "swapping" persists in common usage (e.g., "the system is swapping") but refers to this page-level mechanism, not whole-process swaps.

## 10.8 The Memory Hierarchy Perspective

Memory management does not exist in isolation. It is part of a larger memory hierarchy designed to bridge the speed gap between the processor and storage.

| Level | Technology | Typical Size | Access Time | Cost per GB |
|-------|-----------|-------------|-------------|-------------|
| Registers | SRAM (on-die) | ~1 KB | 0.3 ns | -- |
| L1 Cache | SRAM | 32--64 KB | 1 ns | ~$1000 |
| L2 Cache | SRAM | 256 KB--1 MB | 3--5 ns | ~$100 |
| L3 Cache | SRAM | 8--64 MB | 10--20 ns | ~$30 |
| Main Memory | DRAM | 8--256 GB | 50--100 ns | ~$3 |
| SSD | NAND Flash | 0.5--8 TB | 10--100 $\mu$s | ~$0.10 |
| HDD | Magnetic | 1--20 TB | 5--10 ms | ~$0.02 |

The operating system's memory management subsystem operates primarily at the DRAM level, managing the allocation of physical frames to processes. Caching is handled by hardware (L1--L3) and by the OS page cache (for file data). Swapping extends the effective memory to include SSD/HDD at a dramatic performance cost.

::: definition
**Definition 10.23 (Effective Access Time with Swapping).** If a fraction $p$ of memory accesses cause a page to be swapped in (the *swap rate*), the effective access time is:

$$\text{EAT} = (1 - p) \times t_{\text{mem}} + p \times t_{\text{swap}}$$

where $t_{\text{mem}}$ is the memory access time and $t_{\text{swap}}$ is the time to service a swap (including disk I/O and context switching).
:::

::: example
**Example 10.23 (EAT with Swapping).** $t_{\text{mem}} = 100$ ns, $t_{\text{swap}} = 10$ ms $= 10{,}000{,}000$ ns, $p = 0.001$ (one in a thousand accesses causes a swap).

$$\text{EAT} = 0.999 \times 100 + 0.001 \times 10{,}000{,}000 = 99.9 + 10{,}000 = 10{,}099.9 \text{ ns}$$

Even with a swap rate of just 0.1%, the effective access time increases by a factor of 100. This illustrates why swap avoidance is critical for performance.
:::

::: example
**Example 10.24 (EAT with SSD Swap).** Same scenario but with NVMe SSD: $t_{\text{swap}} = 80\ \mu\text{s} = 80{,}000$ ns.

$$\text{EAT} = 0.999 \times 100 + 0.001 \times 80{,}000 = 99.9 + 80 = 179.9 \text{ ns}$$

Slowdown factor: $179.9 / 100 = 1.8\times$. Significantly better than HDD, but still a 80% performance penalty at a 0.1% swap rate.
:::

## 10.9 Memory Layout of a Process

Understanding what a process's memory looks like is essential for both operating system designers and application programmers. A typical process on a modern Unix-like system has the following layout in its logical address space:

```text
High address
+-------------------+
|     Stack         |  (grows downward)
|        |          |
|        v          |
+-------------------+
|                   |
|  (unmapped gap)   |
|                   |
+-------------------+
|        ^          |
|        |          |
|     Heap          |  (grows upward)
+-------------------+
|     BSS           |  (uninitialised globals, zero-filled)
+-------------------+
|     Data          |  (initialised globals)
+-------------------+
|     Text          |  (executable code, read-only)
+-------------------+
Low address
```

::: definition
**Definition 10.24 (Process Memory Segments).** A process's logical address space is divided into *segments*:

- **Text (code) segment:** Contains the executable machine instructions. Typically read-only and shared among processes running the same program.
- **Data segment:** Contains initialised global and static variables. Read-write.
- **BSS segment:** Contains uninitialised global and static variables, zero-filled by the loader. Named after a historical assembler directive (*Block Started by Symbol*). Read-write.
- **Heap:** Dynamically allocated memory (via `malloc`, `new`, etc.), growing upward from the end of BSS.
- **Stack:** Function call frames, local variables, and return addresses, growing downward from a high address.
:::

::: example
**Example 10.25 (Segment Sizes).** A compiled C program has the following segment sizes (from the `size` command):

```text
$ size /usr/bin/ls
   text    data     bss     dec     hex filename
 138744    4784    4832  148360   243a8 /usr/bin/ls
```

The text segment is 135 KB, data is 4.7 KB, and BSS is 4.7 KB. The total static size is about 145 KB. The heap and stack grow dynamically during execution.
:::

::: programmer
**Programmer's Perspective: Linux /proc/PID/maps and Go's Memory Layout.**

On Linux, you can inspect any process's memory layout through the `/proc` filesystem. The file `/proc/PID/maps` lists every mapped region in the process's address space:

```c
/* Compile with: gcc -o maps_demo maps_demo.c */
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>

int global_init = 42;       /* .data segment */
int global_uninit;           /* .bss segment  */

int main(void) {
    int stack_var = 7;       /* stack */
    int *heap_var = malloc(sizeof(int));  /* heap */
    *heap_var = 99;

    printf("Text  (main):    %p\n", (void *)main);
    printf("Data  (g_init):  %p\n", (void *)&global_init);
    printf("BSS   (g_uninit):%p\n", (void *)&global_uninit);
    printf("Heap  (malloc):  %p\n", (void *)heap_var);
    printf("Stack (local):   %p\n", (void *)&stack_var);
    printf("\n--- /proc/%d/maps (excerpt) ---\n", getpid());

    char cmd[64];
    snprintf(cmd, sizeof(cmd), "cat /proc/%d/maps", getpid());
    system(cmd);

    free(heap_var);
    return 0;
}
```

Running this program produces output like:

```text
Text  (main):    0x55a3b2c00149
Data  (g_init):  0x55a3b2e01010
BSS   (g_uninit):0x55a3b2e01014
Heap  (malloc):  0x55a3b40002a0
Stack (local):   0x7ffd4a3e1c5c
```

Notice the enormous gap between the heap (~0x55...) and the stack (~0x7ffd...). This is the unmapped region where memory-mapped files and shared libraries are loaded. The `maps` file shows each region with its address range, permissions, offset, device, inode, and pathname:

```text
55a3b2c00000-55a3b2c01000 r-xp 00000000 08:01 12345  /home/user/maps_demo
55a3b2e01000-55a3b2e02000 rw-p 00001000 08:01 12345  /home/user/maps_demo
55a3b4000000-55a3b4021000 rw-p 00000000 00:00 0      [heap]
7f4a38000000-7f4a381c0000 r-xp 00000000 08:01 67890  /lib/x86_64-linux-gnu/libc-2.31.so
7ffd4a3c0000-7ffd4a3e2000 rw-p 00000000 00:00 0      [stack]
```

The permissions column uses the standard `rwxp`/`rwxs` format: `r` = read, `w` = write, `x` = execute, `p` = private (CoW), `s` = shared.

Go's memory layout differs significantly from C. The Go runtime reserves a large virtual address region at startup and manages it internally. The `runtime.ReadMemStats` function reports heap statistics, and `GODEBUG=gctrace=1` enables garbage collector tracing. Go goroutine stacks are allocated on the heap (not the OS stack), start small (2--8 KB), and grow dynamically by copying. This means that `&variable` in Go may point to an address that changes over time, which is why Go forbids storing Go pointers in C memory when using cgo. You can inspect a Go process's memory layout with:

```go
package main

import (
    "fmt"
    "os"
    "runtime"
)

func main() {
    var m runtime.MemStats
    runtime.ReadMemStats(&m)
    fmt.Printf("HeapAlloc:  %d MB\n", m.HeapAlloc/1024/1024)
    fmt.Printf("HeapSys:    %d MB\n", m.HeapSys/1024/1024)
    fmt.Printf("StackSys:   %d MB\n", m.StackSys/1024/1024)
    fmt.Printf("Sys (total):%d MB\n", m.Sys/1024/1024)

    pid := os.Getpid()
    data, _ := os.ReadFile(fmt.Sprintf("/proc/%d/maps", pid))
    lines := 0
    for i, b := range data {
        if b == '\n' {
            lines++
            if lines >= 15 {
                fmt.Print(string(data[:i+1]))
                break
            }
        }
    }
}
```
:::

## 10.10 Protection and Sharing

Memory management must enforce two orthogonal goals: *protection* (preventing processes from interfering with each other) and *sharing* (allowing processes to access common data efficiently).

### 10.10.1 Protection Mechanisms

The base-limit mechanism provides basic protection: a process cannot access any address outside $[B, B + L)$. More sophisticated protection requires finer granularity:

- **Read/Write/Execute permissions:** Each memory region can be marked as readable, writable, executable, or any combination. The text segment is typically read+execute; the data and heap are read+write; the stack is read+write (and sometimes no-execute for security).
- **Ring-based protection:** x86 processors define four privilege rings (0--3). Ring 0 (kernel) can access all memory; Ring 3 (user) can only access memory permitted by the page tables.
- **Supervisor/user bit:** Each page table entry on x86-64 has a U/S bit. When U/S = 0, only supervisor mode (ring 0) can access the page.

::: definition
**Definition 10.25 (Protection Domain).** A *protection domain* specifies, for each object (memory region, file, device), the set of operations a process can perform on it. The memory management hardware enforces protection domains for memory objects through permission bits in page table entries or segment descriptors.
:::

### 10.10.2 Sharing

When multiple processes run the same program (e.g., multiple instances of a web server), they can share the text segment to save memory. With base-limit registers, sharing is impossible: each process has a single contiguous region. Paging (Chapter 11) solves this by allowing different page table entries in different processes to point to the same physical frame.

::: example
**Example 10.26 (Shared Library Savings).** The GNU C Library (glibc) is approximately 2 MB in size. On a typical Linux system with 200 processes, each using glibc:

- Without sharing: $200 \times 2 \text{ MB} = 400$ MB
- With sharing: $2$ MB (one copy in physical memory, mapped into all 200 address spaces)

Savings: 398 MB. This is why shared libraries exist.
:::

::: example
**Example 10.27 (Shared Libraries on a Real System).** Using `pmap` on a Linux system:

```text
$ pmap -x $(pidof bash) | head -10
Address           Kbytes     RSS   Dirty Mode  Mapping
000055a7e1200000     880     880       0 r-x-- bash
000055a7e14dc000      16      16      16 r---- bash
000055a7e14e0000      36      36      36 rw--- bash
00007f9c42600000    1820    1200       0 r-x-- libc-2.31.so
00007f9c427ca000    2048       0       0 ----- libc-2.31.so
00007f9c429ca000      16      16      16 r---- libc-2.31.so
```

The `r-x--` regions are shared (read-only text segments). The RSS column shows only the resident pages; the actual physical memory is shared among all processes using the same library.
:::

### 10.10.3 Memory Protection Keys (MPK)

Modern x86-64 processors (Skylake and later) support *Memory Protection Keys*, a hardware feature that allows user-space code to change memory permissions without system calls.

::: definition
**Definition 10.26 (Memory Protection Keys).** *Intel MPK* assigns each page a 4-bit protection key (one of 16 domains). A per-thread register (PKRU) specifies, for each key, whether access is disabled or write-disabled. Changing the PKRU register is a user-space instruction (WRPKRU), taking ~20 cycles --- far faster than a system call to `mprotect()`.
:::

MPK is used for security (isolating sensitive data within a single address space) and for performance (fast toggling of write protection for garbage collectors and software transactional memory).

## 10.11 Segmentation

Before paging became dominant, *segmentation* was the primary mechanism for providing logical address spaces with multiple regions of varying sizes.

::: definition
**Definition 10.27 (Segmentation).** *Segmentation* divides a process's logical address space into a collection of *segments*, each with a name (or number), a base address, and a length. A logical address is a pair (segment number, offset). The segment table maps each segment number to a physical base address and a limit.
:::

A logical address in a segmented system has the form $(s, d)$ where $s$ is the segment number and $d$ is the offset within the segment. The hardware translates this as:

1. Look up segment $s$ in the segment table to obtain base $b_s$ and limit $l_s$.
2. Check $d < l_s$. If not, raise a trap.
3. Compute physical address $p = b_s + d$.

::: example
**Example 10.28 (Segment Table).** A process has four segments:

| Segment | Name | Base | Limit |
|---------|------|------|-------|
| 0 | Code | 0x4000 | 0x1000 |
| 1 | Data | 0x6000 | 0x0800 |
| 2 | Heap | 0x8000 | 0x2000 |
| 3 | Stack | 0xF000 | 0x1000 |

Logical address (2, 0x0100) translates to: $d = 0\text{x}0100 < 0\text{x}2000$ (pass), physical address $= 0\text{x}8000 + 0\text{x}0100 = 0\text{x}8100$.

Logical address (1, 0x0900) fails: $0\text{x}0900 \not< 0\text{x}0800$, segmentation fault.
:::

::: example
**Example 10.29 (Sharing via Segmentation).** Two processes, P1 and P2, run the same editor program with different data.

P1's segment table:

| Segment | Base | Limit |
|---------|------|-------|
| 0 (code) | 0x4000 | 0x3000 |
| 1 (data) | 0xA000 | 0x2000 |

P2's segment table:

| Segment | Base | Limit |
|---------|------|-------|
| 0 (code) | 0x4000 | 0x3000 |
| 1 (data) | 0xD000 | 0x1800 |

Both processes share segment 0 (same base), which contains the editor code. Each has its own data segment. This saves 0x3000 bytes (12 KB) of physical memory.
:::

Segmentation provides a natural model of program structure (code, data, stack are separate segments) and supports sharing (two processes can share a segment by having the same base in their segment tables). However, segmentation suffers from external fragmentation (segments are variable-sized contiguous blocks) and has largely been superseded by paging and paged segmentation.

> **Note:** The Intel x86 architecture historically supported both segmentation and paging. In 64-bit long mode (x86-64), segmentation is largely disabled: the segment base is forced to zero for code and data segments, effectively giving each process a flat address space. Only the FS and GS segment registers retain non-zero bases, used by the OS for per-CPU data and by user space for thread-local storage (TLS).

## 10.12 Paged Segmentation

Some systems combine segmentation and paging to get the logical benefits of both.

::: definition
**Definition 10.28 (Paged Segmentation).** In *paged segmentation*, each segment is divided into fixed-size pages. A logical address is a triple (segment number, page number, offset). The segment table entry for segment $s$ points to a page table for that segment, and the page table maps pages to physical frames.
:::

This hybrid approach eliminates external fragmentation (because pages are fixed-size) while preserving the logical structure of segmentation (separate code, data, stack segments with independent protection bits). The Intel 386 through Pentium 4 (in 32-bit mode) used this scheme.

The address translation for paged segmentation proceeds in four steps:

1. Use the segment number to index the segment table, obtaining the page table base address and segment limit.
2. Check that the (page number, offset) does not exceed the segment limit.
3. Use the page number to index the segment's page table, obtaining the physical frame number.
4. Concatenate the frame number with the offset to produce the physical address.

::: example
**Example 10.30 (Paged Segmentation Translation).** A system has 4 segments, 256 pages per segment, and 4 KB pages. A logical address is (segment=2, page=10, offset=0x0A0).

1. Segment table[2] gives: page table base at physical address 0x50000, limit = 200 pages.
2. Check: page 10 < 200 (pass).
3. Page table at 0x50000, entry 10 gives: frame number = 0x3A.
4. Physical address: $0\text{x}3\text{A} \times 0\text{x}1000 + 0\text{x}0\text{A}0 = 0\text{x}3\text{A}0\text{A}0$.
:::

## 10.13 Historical Context

Memory management has evolved dramatically over the history of computing:

**1950s (No OS):** Programs were loaded at fixed addresses. The programmer managed memory manually.

**1960s (Batch processing):** Simple partitioning schemes (fixed partitions) allowed multiple jobs to reside in memory. IBM's OS/360 introduced relocatable code.

**1964 (Multics):** The first system to use segmentation with paging. Multics provided a flat, single-level virtual address space across all processes.

**1970s (Unix):** The original Unix used swapping (whole-process swap). The VAX-11/780 (1978) introduced demand paging to Unix (BSD).

**1985 (Intel 80386):** Hardware paging support in a mainstream processor, combining segmentation and paging.

**1990s--2000s:** Virtual memory with demand paging became universal. 64-bit address spaces (Alpha, SPARC64, x86-64) removed the 4 GB barrier.

**2010s--present:** Focus shifted to performance: huge pages, TLB optimisation, NUMA awareness, memory compression, and hardware support for memory protection (MPK, CET).

## 10.14 Summary

This chapter has established the fundamental concepts of memory management:

- **Logical vs physical addresses:** Programs use logical addresses; the MMU translates them to physical addresses.
- **Address binding** can occur at compile time, load time, or execution time, with increasing flexibility.
- **Base-limit registers** provide the simplest hardware-supported address translation and protection.
- **Contiguous allocation** (fixed and variable partitions) assigns each process a single block of physical memory.
- **Allocation strategies** (first-fit, best-fit, worst-fit, next-fit) determine which free hole satisfies a request, with first-fit generally preferred.
- **Fragmentation** (internal and external) is the inevitable cost of memory allocation; compaction is the brute-force remedy but is costly and NP-hard to optimise.
- **Swapping** moves entire processes between memory and disk, but its coarse granularity makes it inefficient.
- **The memory hierarchy** spans registers to disk, with access times ranging from 0.3 ns to 10 ms, a factor of $10^7$.
- **Segmentation** provides a structured logical address space but suffers from external fragmentation.
- **Paged segmentation** combines the structure of segmentation with the fragmentation-free property of paging.

The critical limitation of all schemes in this chapter is the requirement for *contiguous* physical allocation (or contiguous segments). Chapter 11 introduces *virtual memory* with *paging*, which shatters this constraint by allowing a process's pages to be scattered anywhere in physical memory, dramatically improving both memory utilisation and the sophistication of memory management.
